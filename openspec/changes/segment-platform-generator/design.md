## Context

当前 `PlatformGenerator.cs` 使用逐块独立随机生成平台行。在微信小游戏竖屏 (9:16) 下，`blockSize=1` 导致每行仅 4~5 块砖块，且空隙/砖块分布呈"椒盐噪声"式零散排列。需要重构为段落式生成算法，缩小砖块尺寸，并明确可破碎/不可破碎/空隙三种类型。

现有代码结构：
- `PlatformGenerator.cs` — 平台生成、移动、回收（核心改动目标）
- `Ground.cs` — 踩踏溶解逻辑（保留，增加攻击响应）
- `BreakableBlock.cs` — 攻击碎裂逻辑（将整合到 Ground.cs）
- `MapPool.cs` / `MapRow.cs` / `RowsGenerate.cs` — 已弃用的旧系统

## Goals / Non-Goals

**Goals:**
- 实现段落式行生成算法，产出类似 `[1,1,2,1,1,2,2,2,1,1,1,3,3,3,...]` 的连续序列
- 砖块尺寸适配竖屏，一行可容纳 10~12 块
- 行间空隙防完全对齐，确保玩家总有路走
- 明确三种砖块类型的行为边界
- 预留难度曲线参数接口

**Non-Goals:**
- 不做关卡编辑器或手动关卡设计
- 不改变平台向上移动的核心玩法
- 不改变玩家控制器逻辑
- 不做多层平台或立体结构

## Decisions

### D1: 段落式生成 vs 噪声/波函数坍缩

**选择**: 段落式顺序生成（逐段填充一行）

**理由**: 
- 规则简单直观，容易调参
- 不需要回溯，O(n) 生成
- 段长度和类型权重直接映射到游戏体验
- 波函数坍缩对此场景过于复杂

**实现要点**:
```
生成一行 = 循环 {选段类型(权重) → 选段长度(范围) → 填入序列} 直到填满
约束后处理：行首行尾不可为空隙；连续空隙 ≤ maxGapLength
```

### D2: 行间验证策略 — 轻量偏移 vs 全局路径验证

**选择**: 轻量偏移检测

**理由**:
- 只需记录上一行的空隙区间列表
- 新行生成后检查：如果新空隙完全被上一行空隙包含，则随机偏移新行空隙
- 不需要复杂的路径搜索算法
- "不完全对齐就行" 的需求不需要严格的可达性证明

### D3: 砖块类型体系 — 合并 vs 分离脚本

**选择**: 分离脚本

**理由**:
- `Ground.cs` → 改名语义为"可破碎砖块"，保留踩踏溶解 + 新增 `OnTriggerEnter2D` 对 hitBox 的攻击响应
- 新建 `SolidBlock.cs` → 不可破碎砖块，纯 MonoBehaviour + Collider2D，无任何消失逻辑
- `BreakableBlock.cs` → 攻击碎裂逻辑整合到 `Ground.cs` 后，此脚本标记弃用
- 分离脚本避免条件分支膨胀，每种砖块职责清晰

### D4: 砖块尺寸策略

**选择**: `blockSize` 默认改为 0.5，但保持为可配置参数

**计算**: 
- 竖屏 9:16, orthographicSize=5 → 水平范围 ±2.8125 → 可用宽度 ≈ 5.0 (扣边距)
- blockSize=0.5, spacing=0.02 → 每块占 0.52 → 一行约 9~10 块
- 这与用户期望的 10+ 块吻合

### D5: 对象池策略

**选择**: 保留现有对象池框架，但重构池化回收逻辑

**理由**: 
- 现有 `platformPool` 缓存整个平台 GameObject（含子砖块）
- 回收后重新生成时，需要清除旧子砖块、重新按新序列生成
- 简化为：回收时销毁所有子砖块，复用父容器 GameObject；重新生成时按新序列添加子砖块
- 砖块本身的 Prefab 可以用独立的砖块池管理（后续优化）

## Risks / Trade-offs

- **[风险] 砖块尺寸与 Sprite 资源不匹配** → 需要确认 Prefab 的 SpriteRenderer 在 scale=0.5 时视觉效果可接受，或需要新的小尺寸 Sprite
- **[风险] 段落式随机可能产生重复感** → 通过足够多的段长度变化和难度曲线缓解
- **[取舍] 对象池简化导致更多 Instantiate/Destroy** → 短期可接受，后续可优化为砖块级别池化
- **[取舍] 行间只做轻量偏移而非严格可达性验证** → 极端情况下可能出现难以通过的区域，但概率很低
